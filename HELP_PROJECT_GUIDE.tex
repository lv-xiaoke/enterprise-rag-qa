\section{1. 项目一句话说明}

这是一个基于 \texttt{Streamlit + LangChain ReAct Agent + Chroma + DashScope} 的扫地机器人智能客服演示项目。\par

它把用户问题交给 ReAct Agent 判断：如果是产品知识、故障排查、选购建议，就调用 RAG 知识库；如果是个人使用报告，就按固定工具链读取外部用户记录；如果是耗材、维护、保养问题，就生成耗材更换提醒；如果问题涉及天气和环境，还可以调用模拟天气工具辅助回答。\par

\section{2. 项目能做什么}

当前项目主要支持四类能力：\par

\begin{enumerate}
\item 知识库问答
\end{enumerate}
从 \texttt{data/} 目录中的 \texttt{txt/pdf} 文档检索相关内容，再交给大模型总结回答。\par

\begin{enumerate}
\item 工具辅助问答
\end{enumerate}
ReAct Agent 可以根据问题自动选择工具，例如 \texttt{get\_weather}、\texttt{get\_user\_location}、\texttt{get\_user\_id}、\texttt{get\_current\_month}。\par

\begin{enumerate}
\item 个性化使用报告
\end{enumerate}
Agent 按固定顺序获取用户 ID、月份、报告上下文和外部数据，然后生成个人扫地机器人使用报告。\par

\begin{enumerate}
\item 耗材维护提醒
\end{enumerate}
Agent 可以读取用户在指定月份的耗材记录，生成滤网、边刷、主刷、胶刷、尘盒、水箱等维护建议。\par

前端入口是 \texttt{app.py}。启动后会打开一个 Streamlit 聊天页面，页面标题为“智扫通机器人智能客服”。\par

\section{3. 项目主链路}

真实调用链路如下：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
Streamlit(app.py)
  -> ReactAgent(agent/react_agent.py)
  -> AgentExecutor.stream()
  -> Tools(agent/tools/agent_tools.py)
  -> RAG(rag/rag_service.py)
  -> Chroma(rag/vector_store.py)
  -> DashScope(model/factory.py)
  -> Prompt(prompts/*.txt)
\end{Verbatim}

用户在 \texttt{app.py} 的 \texttt{st.chat\_input()} 输入问题后，页面会调用 \texttt{ReactAgent.execute\_stream()}。\texttt{ReactAgent} 内部通过 \texttt{create\_react\_agent()} 创建 ReAct Agent，再用 \texttt{AgentExecutor} 包装执行。最终答案通过 \texttt{AgentExecutor.stream()} 分块返回给 Streamlit 页面，并由 \texttt{st.write\_stream()} 展示。\par

\section{8. 核心知识点直接答案}

\subsection{8.1 什么是 RAG}

RAG 是 Retrieval-Augmented Generation，即“检索增强生成”。它先从外部知识库检索和问题相关的资料，再把这些资料连同用户问题一起交给大模型生成答案。\par

本项目中的 RAG 是：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
用户问题 -> Chroma 检索相关文档 -> 拼接 context -> DashScope 模型总结回答
\end{Verbatim}

它解决的问题是：大模型自身不知道本地文档里的项目资料、产品问答和故障排查内容，所以需要先检索，再回答。\par

\subsection{8.2 为什么要把文档切成 chunk}

因为原始文档可能很长，不能整篇都塞给 Embedding 或聊天模型。切成 chunk 后，每个小片段都可以单独向量化和检索。\par

切分的好处：\par

\begin{itemize}
\item 提高检索粒度：只找最相关的片段。
\item 降低上下文长度：减少传给模型的无关内容。
\item 提升命中率：同一篇文档中不同主题可以分别被检索到。
\end{itemize}

\subsection{8.3 \texttt{chunk\_size}、\texttt{chunk\_overlap}、\texttt{k} 分别影响什么}

\begin{itemize}
\item \texttt{chunk\_size} 控制每个文本块的最大长度。太小容易割裂上下文，太大容易混入无关信息。
\item \texttt{chunk\_overlap} 控制相邻文本块的重叠长度。适当重叠能减少断句导致的信息丢失，但过大容易重复入库。
\item \texttt{k} 控制每次检索返回多少个片段。太小可能漏掉答案，太大可能让模型看到噪声。
\end{itemize}

当前配置是 \texttt{chunk\_size=200}、\texttt{chunk\_overlap=20}、\texttt{k=3}，适合这个中小型中文知识库的演示场景。\par

\subsection{8.4 什么是 Embedding}

Embedding 是把文本转换成向量数字的过程。语义相近的文本，向量距离通常更近。\par

本项目使用 \texttt{DashScopeEmbeddings(model="text-embedding-v4")} 把知识库片段转成向量，然后存入 Chroma。用户提问时，也会被转成向量，再和库里的向量做相似度检索。\par

\subsection{8.5 什么是 Chroma}

Chroma 是一个向量数据库，用来存储文档、向量和元数据，并支持相似度查询。\par

本项目用 \texttt{langchain\_chroma.Chroma} 创建本地向量库：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
Chroma(
    collection_name="agent",
    embedding_function=embed_model,
    persist_directory="chroma_db",
)
\end{Verbatim}

\texttt{persist\_directory="chroma\_db"} 表示向量库会持久化到本地目录，下次启动仍可复用。\par

\subsection{8.6 什么是 Retriever}

Retriever 是检索器。它把“向量数据库怎么查”封装成一个统一接口。\par

本项目中的代码是：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
self.vector_store.as_retriever(search_kwargs={"k": chroma_conf["k"]})
\end{Verbatim}

调用 \texttt{retriever.invoke(query)} 后，会返回和 query 最相关的 \texttt{Document} 列表。\par

\subsection{8.7 什么是 Agent}

Agent 是能“自己决定下一步做什么”的大模型应用。普通 LLM 调用只生成回答，Agent 则可以根据问题选择工具、调用工具、观察结果，再决定是否继续调用工具或输出最终答案。\par

一句话区分：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
普通 RAG：先检索，再回答。
Agent：模型自己判断是否检索、调用哪个工具、调用几次。
\end{Verbatim}

\subsection{8.8 什么是 ReAct}

ReAct 是 Reasoning + Acting，即“推理 + 行动”。模型按如下格式工作：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
Thought -> Action -> Action Input -> Observation -> Final Answer
\end{Verbatim}

在本项目中：\par

\begin{itemize}
\item \texttt{Thought}：模型分析用户需求。
\item \texttt{Action}：选择工具名，例如 \texttt{rag\_summarize}。
\item \texttt{Action Input}：传给工具的参数。
\item \texttt{Observation}：工具返回结果。
\item \texttt{Final Answer}：整合结果后给用户的最终回答。
\end{itemize}

\subsection{8.9 \texttt{create\_react\_agent()} 做什么}

\texttt{create\_react\_agent()} 把聊天模型、工具列表和 ReAct Prompt 组合成一个可运行的 Agent。\par

本项目在 \texttt{agent/react\_agent.py} 中调用：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
agent = create_react_agent(
    llm=chat_model,
    tools=tools,
    prompt=prompt
)
\end{Verbatim}

注意：ReAct Prompt 必须包含 \texttt{tools}、\texttt{tool\_names}、\texttt{agent\_scratchpad} 等变量，否则 Agent 无法知道有哪些工具，也无法记录中间推理和工具观察。\par

\subsection{8.10 \texttt{AgentExecutor} 做什么}

\texttt{AgentExecutor} 负责真正执行 Agent 循环，包括：\par

\begin{itemize}
\item 把用户输入交给 Agent。
\item 解析模型输出中的 \texttt{Action} 和 \texttt{Action Input}。
\item 调用对应工具。
\item 把工具结果作为 \texttt{Observation} 交回模型。
\item 循环直到模型输出 \texttt{Final Answer}。
\end{itemize}

本项目还配置了 \texttt{handle\_parsing\_errors}，当模型输出格式不符合 ReAct 要求时，给模型一个纠错提示，减少前端崩溃。\par

\subsection{8.11 \texttt{@tool} 装饰器有什么用}

\texttt{@tool} 会把普通 Python 函数包装成 LangChain Tool。函数名会成为工具名，函数 docstring 会成为工具描述，模型会根据工具描述判断什么时候调用它。\par

例如：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
@tool
def get_weather(city: str) -> str:
    """获取指定城市的天气，以消息字符串的形式返回"""
    return "..."
\end{Verbatim}

所以新增工具时，docstring 要写清楚“工具能做什么、什么时候用、入参格式是什么”。\par

\subsection{8.12 Streamlit 在项目中负责什么}

Streamlit 负责前端聊天界面：\par

\begin{itemize}
\item \texttt{st.title()} 显示页面标题。
\item \texttt{st.chat\_input()} 接收用户输入。
\item \texttt{st.chat\_message()} 展示用户和助手消息。
\item \texttt{st.session\_state} 保存聊天历史和 Agent 实例。
\item \texttt{st.write\_stream()} 流式展示模型答案。
\end{itemize}

\subsection{8.13 为什么要用 YAML 配置}

YAML 把模型名、Prompt 路径、Chroma 参数、外部数据路径从代码中抽离出来。这样调整模型、切分参数、知识库路径时不需要改核心代码。\par

当前配置分工：\par

\begin{itemize}
\item \texttt{rag.yml} 管模型。
\item \texttt{chroma.yml} 管向量库和知识库切分。
\item \texttt{prompts.yml} 管 Prompt 文件路径。
\item \texttt{agent.yml} 管外部业务数据路径。
\end{itemize}

\section{9. 知识库构建与更新}

如果 \texttt{chroma\_db/} 中已有可用向量库，可以直接运行应用。\par

如果新增或修改了 \texttt{data/} 中的知识库文档，需要重新执行入库逻辑：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
python -m rag.vector_store
\end{Verbatim}

入库流程在 \texttt{rag/vector\_store.py}：\par

\begin{enumerate}
\item 扫描 \texttt{config/chroma.yml} 中 \texttt{data\_path} 指向的目录。
\item 只读取 \texttt{txt/pdf} 文件。
\item 计算文件 MD5，跳过已经处理过的文件。
\item 用 \texttt{TextLoader} 或 \texttt{PyPDFLoader} 加载文档。
\item 用 \texttt{RecursiveCharacterTextSplitter} 切分文档。
\item 用 DashScope Embedding 生成向量。
\item 写入 Chroma。
\item 将已处理文件 MD5 写入 \texttt{md5.txt}。
\end{enumerate}

注意：如果想完全重建向量库，需要手动确认旧的 \texttt{chroma\_db/} 和 \texttt{md5.txt}。不要在脚本中批量删除目录。\par

\section{12. 工作流程详解}

\subsection{12.1 普通知识库问答}

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
用户输入产品问题
  -> Streamlit 调用 ReactAgent.execute_stream()
  -> Agent 判断需要专业资料
  -> 调用 rag_summarize
  -> RagSummarizeService 检索 Chroma
  -> 拼接参考资料 context
  -> DashScope 模型基于参考资料生成回答
  -> Streamlit 流式展示答案
\end{Verbatim}

\subsection{12.2 使用报告生成}

报告生成在 \texttt{agent/react\_agent.py} 的 Prompt 中被强约束为固定顺序：\par

\begin{enumerate}
\item 调用 \texttt{get\_user\_id} 获取用户 ID。
\item 调用 \texttt{get\_current\_month} 获取月份。
\item 调用 \texttt{fill\_context\_for\_report} 注入报告场景上下文。
\item 调用 \texttt{fetch\_external\_data}，传入 \texttt{"用户ID,月份"}。
\item 根据外部数据生成最终报告。
\end{enumerate}

当前 \texttt{get\_user\_id} 和 \texttt{get\_current\_month} 使用随机示例数据，所以每次生成报告的用户和月份可能不同。\par

\subsection{12.3 耗材维护提醒}

耗材维护提醒使用 \texttt{generate\_consumable\_reminder(params\_str)} 工具。\par

调用格式：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
用户ID,月份
\end{Verbatim}

示例：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
1001,2025-03
\end{Verbatim}

如果用户没有明确给出用户 ID 或月份，Agent 会先调用 \texttt{get\_user\_id} 和 \texttt{get\_current\_month}，再把结果拼成 \texttt{"用户ID,月份"} 传入工具。\par

工具会返回：\par

\begin{itemize}
\item 用户画像。
\item 清洁表现。
\item 耗材状态。
\item 同类对比。
\item 维护建议。
\end{itemize}

\section{13. 主要代码入口}

\subsection{13.1 \texttt{app.py}}

负责 Streamlit 页面：\par

\begin{itemize}
\item 初始化 \texttt{ReactAgent}。
\item 保存聊天历史到 \texttt{st.session\_state["message"]}。
\item 接收用户输入。
\item 调用 \texttt{execute\_stream()} 获取流式响应。
\item 用 \texttt{st.write\_stream()} 展示答案。
\end{itemize}

注意点：当前代码最后保存助手消息时使用 \texttt{response\_messages[-1]}，这通常只保存最后一个流式 chunk。如果以后要完整保存整段回复，可改为 \texttt{"".join(response\_messages)}。\par

\subsection{13.2 \texttt{agent/react\_agent.py}}

负责构建 ReAct Agent：\par

\begin{itemize}
\item 加载主 Prompt。
\item 注册工具列表。
\item 拼接 ReAct 标准模板。
\item 使用 \texttt{create\_react\_agent()} 创建 Agent。
\item 使用 \texttt{AgentExecutor} 包装 Agent。
\item 通过 \texttt{execute\_stream()} 提供流式输出。
\end{itemize}

当前工具列表：\par

tools = [

    get_weather,

    get_user_id,

    fetch_external_data,
    fill_context_for_report,
    generate_consumable_reminder,
]
\end{Verbatim}

\subsection{13.3 \texttt{agent/tools/agent\_tools.py}}

定义 Agent 可调用工具：\par

\begin{itemize}
\item \texttt{rag\_summarize}：知识库检索问答。
\item \texttt{get\_weather}：返回模拟天气。
\item \texttt{get\_user\_location}：随机返回用户城市。
\item \texttt{get\_user\_id}：随机返回用户 ID。

\item \texttt{fetch\_external\_data}：读取 \texttt{records.csv} 中的外部记录。

负责知识库入库和检索器创建：\par

python -m rag.vector_store

\begin{enumerate}
\item 启动应用并提问验证。
\end{enumerate}

\end{itemize}

\section{15. 如何新增 Agent 工具}

新增工具通常改两处：\texttt{agent/tools/agent\_tools.py} 和 \texttt{agent/react\_agent.py}。\par

第一步，在 \texttt{agent/tools/agent\_tools.py} 中定义工具：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
from langchain_core.tools import tool

@tool
def get_order_status(order_id: str) -> str:
    """根据订单号查询订单状态。调用时传入订单号字符串。"""
    return "订单已完成"
\end{Verbatim}

第二步，在 \texttt{agent/react\_agent.py} 中导入并加入工具列表：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
tools = [
    rag_summarize,
    get_weather,
    get_order_status,
]
\end{Verbatim}

第三步，在 Prompt 中说明工具调用场景和入参格式。工具 docstring 和 Prompt 说明越清楚，Agent 越容易正确调用。\par

\section{16. 常见问题}

\subsection{16.1 启动时报 DashScope API Key 错误}

检查是否设置了环境变量：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
$env:DASHSCOPE_API_KEY
\end{Verbatim}

如果没有输出，重新设置：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
$env:DASHSCOPE_API_KEY="your-api-key"
\end{Verbatim}

\subsection{16.2 问答结果和知识库内容无关}

检查：\par

\begin{itemize}
\item \texttt{chroma\_db/} 是否已经生成。
\item 是否执行过 \texttt{python -m rag.vector\_store}。
\item \texttt{data/} 中是否有相关文档。
\item \texttt{config/chroma.yml} 中 \texttt{k} 是否太小。
\item 文档编码是否为 UTF-8。
\item \texttt{rag\_summarize.txt} 是否仍要求模型基于参考资料回答。
\end{itemize}

\subsection{16.3 新增文档后没有被检索到}

项目用 MD5 记录已处理文件。如果文档已经入库，再次运行时会跳过相同 MD5 的文件。可以查看 \texttt{logs/} 中的日志确认是否跳过。\par

如果确实需要完全重建，请手动确认后处理旧向量库和 \texttt{md5.txt}，不要使用批量删除命令。\par

\subsection{16.4 Agent 输出格式错误或中断}

ReAct Agent 对输出格式要求较严格，模型需要生成：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
Thought / Action / Action Input / Observation / Final Answer
\end{Verbatim}

项目已在 \texttt{AgentExecutor} 中配置 \texttt{handle\_parsing\_errors}。如果仍失败，可以：\par

\begin{itemize}
\item 换一种更明确的问题问法。
\item 检查 \texttt{prompts/main\_prompt.txt} 是否被改坏。
\item 换用更稳定的聊天模型。
\item 减少 Prompt 中和 ReAct 标准格式冲突的说明。
\end{itemize}

\subsection{16.5 Streamlit 页面聊天历史不完整}

当前 \texttt{app.py} 保存助手回复时使用 \texttt{response\_messages[-1]}，如果流式输出被切成多个 chunk，历史里可能只保存最后一段。可改为：\par

\begin{Verbatim}[breaklines=true,breakanywhere=true,fontsize=\small]
st.session_state["message"].append({
    "role": "assistant",
    "content": "".join(response_messages),
})
\end{Verbatim}

\subsection{16.6 README 或终端显示乱码}

中文文档在部分 Windows 终端中可能因编码显示异常。建议使用支持 UTF-8 的编辑器查看 Markdown，并优先参考本 \texttt{HELP.md}。\par

\section{19. 验收问题与答案}

\subsection{RAG 为什么需要把文档切成 chunk}

因为文档通常很长，直接整体检索粒度太粗，也容易超过模型上下文。chunk 可以让系统按小片段做向量检索，提高相关性并减少噪声。\par

\subsection{\texttt{chunk\_size}、\texttt{chunk\_overlap}、\texttt{k} 分别影响什么}

\texttt{chunk\_size} 影响片段长度，\texttt{chunk\_overlap} 影响相邻片段的上下文连续性，\texttt{k} 影响每次检索返回多少片段。\par

\subsection{\texttt{rag\_summarize} 什么时候会被 Agent 调用}

当用户询问扫地机器人知识库问题、故障排查、选购建议、使用方法、维护保养等专业问题时，Agent 应优先调用 \texttt{rag\_summarize}。\par

\subsection{\texttt{AgentExecutor} 和普通 LLM 调用有什么区别}

普通 LLM 调用只输入问题、输出回答。\texttt{AgentExecutor} 会执行 Agent 循环，支持模型选择工具、调用工具、读取观察结果并继续推理。\par

\subsection{新增一个工具需要改哪些地方}

通常需要：\par

\begin{enumerate}
\item 在 \texttt{agent/tools/agent\_tools.py} 中用 \texttt{@tool} 定义函数。
\item 在 \texttt{agent/react\_agent.py} 中导入并加入 \texttt{tools} 列表。
\item 在 Prompt 中补充调用场景和入参格式。
\end{enumerate}
