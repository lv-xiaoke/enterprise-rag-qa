这是一个企业知识库智能助手，基于 LangChain + ReAct Agent + Chroma 向量数据库构建。 它可以回答企业制度问题、查询员工模拟数据、自动生成周报，并通过 Streamlit 提供流式对话界面。

2. 核心概念：LangChain 框架

2.1 LangChain 是什么？

LangChain 是一个用于构建 LLM（大语言模型）驱动应用的开发框架。 它的核心思想是将 LLM 与外部的数据源、工具、记忆等组件连接起来，形成可编排的"链"（Chain）和"智能体"（Agent）。

2.2 本项目使用的 LangChain 核心组件

2.2.1 ChatModel（聊天模型）

ChatModel 是 LangChain 对 LLM 的统一抽象。本项目中使用了阿里通义千问的 ChatTongyi：

from langchain_community.chat_models.tongyi import ChatTongyi

chat_model = ChatTongyi(model="qwen-max")
原理说明
ChatTongyi 封装了阿里 DashScope API，实现了 LangChain 的 BaseChatModel 接口。这意味着它可以被 LangChain 的任何 Agent、Chain 直接使用，而不需要关心底层 API 细节。

2.2.2 Embeddings（嵌入模型）

Embedding 模型将文本转换为高维向量（一组浮点数），语义相近的文本向量距离也相近：

from langchain_community.embeddings import DashScopeEmbeddings

embed_model = DashScopeEmbeddings(model="text-embedding-v4")

2.2.3 Document（文档对象）

LangChain 中的 Document 是一个标准化的数据结构，包含两个核心字段：

from langchain_core.documents import Document

# Document 的结构
Document(
    page_content="实际文本内容...",  # 文本正文
    metadata={"source": "file.pdf", "page": 3}  # 元数据
)

2.2.4 PromptTemplate（提示词模板）

支持变量插值的提示词模板，将动态数据注入固定结构：

from langchain_core.prompts import PromptTemplate

template = PromptTemplate.from_template(
    "基于以下资料回答问题：{context}\n问题：{input}"
)
# 调用时传入变量
result = template.invoke({"context": "...", "input": "..."})

2.2.5 StrOutputParser（字符串输出解析器）

最简单的输出解析器，将 LLM 返回的 AIMessage 对象提取为纯字符串：

from langchain_core.output_parsers import StrOutputParser
# LLM 返回 AIMessage(content="hello") → StrOutputParser → "hello"

2.2.6 LCEL（LangChain Expression Language）

使用 |（管道操作符）连接组件，形成可执行链：

chain = prompt_template | chat_model | StrOutputParser()
# 数据流动：prompt → LLM → 字符串
result = chain.invoke({"input": "什么是RAG？"})
LCEL 的关键优势
LCEL 链式调用是 LangChain 最优雅的设计之一。它自动处理流式输出（stream）、异步调用（async）、批处理（batch）等场景，无需手动编写这些逻辑。

3. 核心概念：ReAct Agent 模式

3.1 ReAct = Reasoning + Acting

ReAct（Reasoning + Acting）是 2022 年由 Google Research 和普林斯顿大学提出的论文模式。 核心思想是让 LLM 交替进行"思考"和"行动"，通过调用外部工具获取信息，从而解决仅凭模型自身知识无法准确回答的问题。

3.2 ReAct 循环公式

Question: 用户输入的问题
Thought: 我应该先做什么？
Action: 要调用的工具名称
Action Input: 传给工具的参数
Observation: 工具返回的结果
  ↓ (重复 N 次，直到有足够信息)
Thought: 我现在有足够的信息回答问题了
Final Answer: 最终回答

3.3 实际执行示例

用户问："我还有多少年假？"

=== 第 1 轮 ===
Thought: 用户想知道自己的年假余额。我需要先获取员工ID，再查询年假数据。
Action: get_employee_id
Action Input: ""
Observation: E1003

=== 第 2 轮 ===
Thought: 已获取员工ID为E1003，现在查询年假余额。
Action: fetch_employee_records
Action Input: E1003,leave_balance
Observation: 剩余年假：12 天。

=== 第 3 轮 ===
Thought: 我已获得年假数据，可以给出最终答案。
Final Answer: 根据系统记录，您当前剩余年假为 12 天。

3.4 create_react_agent 的内部机制

本项目使用 LangGraph 提供的 create_react_agent 函数。其内部原理：

  1. 将工具描述注入 Prompt：每个 @tool 的 docstring 被自动提取为工具描述，嵌入到 ReAct 模板中
  2. 构建 LangGraph 状态图：创建一个包含 "agent" 和 "tools" 两个节点的有向图
  3. Agent 节点：调用 LLM，根据 Prompt 和历史消息生成下一步的 Action 或 Final Answer
  4. Tools 节点：解析 Agent 的输出，如果包含 Action 则执行对应工具，将结果作为 Observation 追加到消息历史
  5. 循环条件：如果 LLM 输出了 Final Answer，则终止循环；否则继续回到 Agent 节点
create_react_agent vs 手写 ReAct
create_react_agent 是 LangGraph 对 ReAct 模式的高度封装。它内部处理了：状态管理（AgentState）、工具路由、解析错误处理、中间件集成、流式输出等。如果手写，需要数百行代码。

3.5 AgentExecutor

AgentExecutor 是对 Agent 的包装，提供了额外的控制能力：

  • verbose=True：打印详细的执行日志
  • handle_parsing_errors：当 LLM 输出格式不符合 ReAct 规范时，向 LLM 发送修正提示，而不是直接崩溃
  • stream()：流式返回执行结果，实现打字机效果

4. 核心概念：RAG 检索增强生成

4.1 RAG 是什么？

RAG（Retrieval-Augmented Generation，检索增强生成）解决 LLM 的两大痛点： 知识截止日期（训练数据过时）和幻觉问题（编造不存在的信息）。 核心流程是：先检索相关文档，再将文档作为上下文注入 Prompt，让 LLM 基于参考资料回答。

4.2 RAG 的五个阶段

阶段 1：文档加载（Document Loading）

from langchain_community.document_loaders import PyPDFLoader, TextLoader

# 加载 PDF
documents = PyPDFLoader("企业制度.pdf").load()
# 加载 TXT
documents = TextLoader("IT规范.txt", encoding="utf-8").load()

阶段 2：文本切分（Text Splitting）

大文档无法直接送入 LLM（有上下文窗口限制），需要切分为小块（chunk）。本项目使用递归字符分割器：

RecursiveCharacterTextSplitter(
    chunk_size=200,       # 每块最多 200 个字符
    chunk_overlap=20,      # 相邻块重叠 20 个字符（保持语义连续性）
    separators=["\n\n", "。", ".", "？", " ", ""],  # 按优先级分割
    length_function=len,
)
为什么需要 chunk_overlap？
假设一段关键信息恰好被切分边界断开（前半段在 chunk A，后半段在 chunk B），overlap 确保相邻 chunk 中都有这段信息的上下文，提高检索召回率。

阶段 3：向量化（Embedding）

将文本块通过 Embedding 模型转换为向量（如 1536 维浮点数数组）：

text = "差旅报销需要发票原件和出差申请单"
vector = embed_model.embed_query(text)
# vector = [0.023, -0.145, 0.892, ..., 0.034]  # 1536 维

阶段 4：存储与检索（Store & Retrieve）

向量存入 Chroma，查询时通过余弦相似度找到最相关的 top-k 个文本块：

retriever = vector_store.as_retriever(search_kwargs={"k": 3})
relevant_docs = retriever.invoke("小户型适合哪些扫地机器人")
# 返回最相关的 3 个文档块

阶段 5：增强生成（Augmented Generation）

将检索结果拼接为上下文，注入 Prompt，交给 LLM 生成回答：

context = ""
for i, doc in enumerate(relevant_docs):
    context += f"【参考资料{i+1}】{doc.page_content}\n"

prompt = f"参考资料：{context}\n问题：{query}\n请基于参考资料回答。"
answer = chat_model.invoke(prompt)

5. 核心概念：Middleware 中间件系统

5.1 什么是中间件？

LangGraph 的中间件系统允许在 Agent 执行的关键节点插入自定义逻辑，类似于 Web 框架（如 Express.js、Django）中的中间件概念。 中间件可以拦截、修改、记录或增强 Agent 的行为，而不需要修改 Agent 本身的核心代码。

5.2 三种中间件类型

装饰器触发时机作用典型用途
@wrap_tool_call 每次工具调用前后 包装工具调用过程 日志记录、性能监控、权限校验、结果缓存
@before_model 每次 LLM 调用之前 观察状态快照 记录状态、消息计数、成本追踪
@dynamic_prompt 每次生成 Prompt 之前 动态切换 Prompt 根据上下文切换系统提示词（如普通模式→报告模式）

5.3 中间件执行流程

@wrap_tool_call @before_model @dynamic_prompt │ │ │ ▼ ▼ ▼ ┌─────────┐ tool_call ┌──────────┐ model_call ┌──────────────┐ │ 工具调用 │◄──────────►│ LLM 调用 │◄────────────►│ Prompt 生成 │ └─────────┘ └──────────┘ └──────────────┘ │ │ │ ▼ ▼ ▼ 记录工具名+参数 记录消息数量+类型 根据 context 切换 记录成功/失败 最后一次消息内容 system prompt

5.4 本项目中的实际应用

最关键的设计是 @dynamic_prompt——它实现了运行时 Prompt 切换：

# 正常运行：使用 main_prompt（企业助手角色）
↓
# 当工具 fill_context_for_report 被调用时（在 monitor_tool 中检测）
request.runtime.context["report"] = True
↓
# 下一次 @dynamic_prompt 触发时
if is_report:
    return load_report_prompts()  # 切换到周报生成 Prompt
为什么用中间件而不是 if-else？
中间件将横切关注点（cross-cutting concerns）从业务逻辑中解耦。监控、日志、Prompt 切换都不需要侵入 ReAct Agent 或工具函数的主体代码。

6. 核心概念：LangGraph 运行时

6.1 LangGraph 在项目中的角色

LangGraph 是一个用于构建有状态、多角色 Agent 应用的框架。 它将 Agent 的执行建模为有向图（节点 = 操作，边 = 数据流），支持条件分支和循环。 本项目中，create_react_agent 底层正是用 LangGraph 构建的。

6.2 AgentState（Agent 状态）

AgentState 是整个 Agent 执行过程中的核心状态对象，包含：

AgentState = {
    "messages": [       # 完整的消息历史列表
        HumanMessage("我还有多少年假？"),
        AIMessage(content="..."),      # LLM 的 Thought/Action
        ToolMessage(content="剩余年假：12天"),  # 工具返回的 Observation
        AIMessage(content="您当前剩余年假12天"),   # Final Answer
    ]
}

6.3 Runtime（运行时上下文）

Runtime 提供了一个跨轮次的上下文存储，允许在中间件之间传递状态：

# 在 monitor_tool 中设置标记
request.runtime.context["report"] = True

# 在 report_prompt_switch 中读取标记
is_report = request.runtime.context.get("report", False)

这种设计使得工具调用可以影响后续的 Prompt 选择，而不需要通过全局变量。

6.4 Command 和 ToolCallRequest

  • ToolCallRequest：封装了单次工具调用的完整信息（工具名、参数、运行时上下文）
  • ToolMessage：工具调用的返回结果，被追加到消息历史中
  • Command：可以中断或重定向 Agent 执行流的特殊返回值
LangGraph vs LangChain 的 AgentExecutor
LangGraph 是 LangChain 生态的进化方向。传统的 AgentExecutor 是线性的"思考→行动→观察"循环， 而 LangGraph 支持更复杂的图结构：条件分支、并行执行、子图嵌套。但本项目的 create_react_agent 在 LangGraph 0.2.x 版本中仍然保持了简洁的 API。

7. 核心概念：Streamlit 对话界面

7.1 Streamlit 的编程模型

Streamlit 采用"脚本即界面"的范式：每次用户交互（点击、输入）都会从头到尾重新执行整个 Python 脚本。 这意味着不需要手动管理 DOM、路由或前后端通信。可变状态通过 st.session_state 在多次重跑之间保持。

7.2 关键 Streamlit API

API作用
st.chat_message(role, avatar)渲染一条聊天消息气泡
st.chat_input(placeholder)底部固定输入框
st.write_stream(generator)流式显示生成器产生的文本（打字机效果）
st.session_state跨重跑持久化状态（Agent 实例、消息历史）
st.sidebar侧边栏（模型选择、参数调节）
st.expander(label, expanded)可折叠区域（展示检索到的文档块）
st.spinner(text)加载动画
st.rerun()手动触发脚本重跑

7.3 流式输出的实现

实现"打字机"效果的核心代码：

def stream_agent_response(prompt):
    st.session_state.messages.append({"role": "user", "content": prompt})

    with st.chat_message("assistant"):
        with st.spinner("正在检索知识库并生成回答..."):
            stream = st.session_state.agent.execute_stream(prompt)

            def capture():
                for chunk in stream:
                    response_chunks.append(chunk)
                    for char in chunk:
                        time.sleep(0.004)  # 逐字符延迟，模拟打字效果
                        yield char

            st.write_stream(capture())  # Streamlit 自动渲染 yield 的字符

9. 代码详解：ReAct Agent 引擎

agent/react_agent.py 是整个项目的心脏，负责创建和运行 ReAct Agent。

9.1 类的结构

class ReactAgent:
    def __init__(self):
        # ① 注册工具列表
        tools = [rag_summarize, get_employee_id, get_employee_department,
                 fetch_employee_records, generate_weekly_report_context]

        # ② 构建 ReAct Prompt 模板
        base_system_prompt = load_system_prompts()
        react_template = base_system_prompt + REACT_FORMAT_INSTRUCTION

        # ③ 创建 PromptTemplate
        prompt = PromptTemplate.from_template(react_template)

        # ④ 创建 LangGraph ReAct Agent
        agent = create_react_agent(llm=chat_model, tools=tools, prompt=prompt)

        # ⑤ 包装为 AgentExecutor
        self.agent_executor = AgentExecutor(
            agent=agent, tools=tools, verbose=True,
            handle_parsing_errors="请检查输出格式..."
        )

9.2 ReAct 模板中的关键部分

Prompt 中包含的 ReAct 格式指令（来自 react_agent.py 第 28-63 行）：

{tools}          # 自动填充：工具名称 + 描述（来自 @tool 的 docstring）
{tool_names}     # 自动填充：所有工具名列表
{input}          # 用户问题
{agent_scratchpad} # 历史 Thought/Action/Observation 的累积

Prompt 中还嵌入了两个工作流模板：

  • 周报工作流：固定 4 步工具调用顺序（get_employee_id → get_employee_department → generate_weekly_report_context → fetch_employee_records）
  • 员工数据工作流：2 步工具调用（get_employee_id → fetch_employee_records）

9.3 流式执行

def execute_stream(self, query: str):
    input_dict = {"input": query}
    config = {"metadata": {"business_scene": "enterprise_knowledge_assistant"}}

    try:
        for chunk in self.agent_executor.stream(input_dict, config=config):
            if "output" in chunk:
                yield chunk["output"] + "\n"       # 每次 yield 一个输出块
    except RuntimeError:
        yield "\n\n[系统提示：模型输出格式不符合 ReAct 要求...]\n"
错误处理设计
RuntimeError 通常表示 LLM 返回了不符合 ReAct 格式的输出（如缺少 Action 字段）。 项目通过 handle_parsing_errors 在 AgentExecutor 层面给予 LLM 修正机会， 同时在最外层也捕获异常以提供友好的错误提示。

10. 代码详解：工具系统

agent/tools/agent_tools.py 定义了 Agent 可调用的 5 个工具。

10.1 @tool 装饰器的工作原理

@tool 是 LangChain 提供的一个装饰器，它将普通 Python 函数自动注册为 Agent 可用的工具。关键机制：

  • 函数名 → 工具名称（Action 中用到的名字）
  • 函数的 docstring → 工具描述（LLM 根据描述决定调用哪个工具）
  • 函数的参数签名 → 工具的输入 Schema（告诉 LLM 应该传什么参数）
  • 函数的返回值 → Observation（追加到消息历史）

10.2 五个工具详解

工具 1：rag_summarize — 知识库检索

@tool
def rag_summarize(query: str) -> str:
    """从企业知识库中检索制度、流程、IT 支持、项目规范等参考资料。"""
    return rag.rag_summarize(query)  # 委托给 RagSummarizeService

这是最核心的工具——它将用户问题转发给 RAG 服务，RAG 服务从 Chroma 向量数据库检索相关文档，拼接到 Prompt 中，再交给 LLM 总结回答。

11. 代码详解：中间件

agent/tools/middleware.py 定义了 3 个中间件，形成 Agent 的监控、日志和动态提示词系统。

11.1 monitor_tool — 工具调用监控器

@wrap_tool_call
def monitor_tool(
    request: ToolCallRequest,                              # 请求数据封装
    handler: Callable[[ToolCallRequest], ToolMessage | Command],  # 实际执行函数
) -> ToolMessage | Command:
    logger.info(f"[tool monitor]执行工具：{request.tool_call['name']}")
    logger.info(f"[tool monitor]传入参数：{request.tool_call['args']}")

    try:
        result = handler(request)   # ★ 调用真正的工具函数
        logger.info(f"[tool monitor]工具{request.tool_call['name']}调用成功")

        # ★ 关键逻辑：检测到特定工具调用时设置标记
        if request.tool_call['name'] == "fill_context_for_report":
            request.runtime.context["report"] = True

        return result
    except Exception as e:
        logger.error(f"工具{request.tool_call['name']}调用失败，原因：{str(e)}")
        raise e
装饰器模式 + 责任链
handler(request) 的调用是中间件链的关键——handler 可能是下一个中间件，也可能是最终的工具函数本身。 多个 @wrap_tool_call 装饰的函数会形成一个调用链。

11.2 log_before_model — 模型调用前日志

@before_model
def log_before_model(state: AgentState, runtime: Runtime):
    logger.info(f"[log_before_model]即将调用模型，带有{len(state['messages'])}条消息。")
    logger.debug(f"[log_before_model]{type(state['messages'][-1]).__name__} | "
                 f"{state['messages'][-1].content.strip()}")
    return None  # 返回 None 表示不做任何修改

这个中间件只观察不修改——它在每次 LLM 调用前记录当前消息数量和最后一条消息的内容，用于调试。

11.3 report_prompt_switch — 动态提示词切换

@dynamic_prompt
def report_prompt_switch(request: ModelRequest):
    is_report = request.runtime.context.get("report", False)
    if is_report:
        return load_report_prompts()   # 返回周报生成专用 Prompt
    return load_system_prompts()       # 返回默认企业助手 Prompt

这是最精妙的设计——通过 runtime.context 在不同中间件之间传递状态， 实现了无需修改 Agent 核心逻辑的 Prompt 动态切换。

12. 代码详解：RAG 服务层

rag/rag_service.py 是 RAG 的核心，将检索和生成串联为一条 LCEL 链。

12.1 RagSummarizeService 的架构

class RagSummarizeService:
    def __init__(self):
        self.vector_store = VectorStoreService()              # 向量存储
        self.retriever = self.vector_store.get_retriever()    # 检索器
        self.prompt_text = load_rag_prompts()                 # RAG 专用 Prompt
        self.prompt_template = PromptTemplate.from_template(self.prompt_text)
        self.model = chat_model                               # LLM 模型
        self.chain = self._init_chain()                       # 构建 LCEL 链

    def _init_chain(self):
        chain = self.prompt_template | print_prompt | self.model | StrOutputParser()
        return chain

12.2 LCEL 链的数据流

一条 LCEL 链定义了完整的数据处理流水线：

输入 dict{"input": query, "context": ...}
  │
  ▼
PromptTemplate       # 将变量填充到模板 → 生成完整 Prompt 字符串
  │
  ▼
print_prompt         # 调试用：打印完整 Prompt 到控制台
  │
  ▼
chat_model           # 调用通义千问 LLM → 返回 AIMessage
  │
  ▼
StrOutputParser()    # 提取 AIMessage.content → 纯字符串
  │
  ▼
返回字符串

12.3 rag_summarize 主流程

def rag_summarize(self, query: str) -> str:
    # 第 1 步：从 Chroma 检索相关文档
    context_docs = self.retriever_docs(query)

    # 第 2 步：拼接上下文
    context = ""
    for i, doc in enumerate(context_docs):
        context += f"【参考资料{i+1}】: {doc.page_content} | 元数据：{doc.metadata}\n"

    # 第 3 步：通过 LCEL 链生成回答
    return self.chain.invoke({
        "input": query,
        "context": context,
    })
retriever.invoke() 的底层发生了什么？
1. 将 query 通过 embedding 模型转换为向量
2. 在 Chroma 中执行近似最近邻搜索（ANN）
3. 返回余弦相似度最高的 k 个 Document 对象

13. 代码详解：向量存储

rag/vector_store.py 管理文档的生命周期：加载→切分→向量化→存储。

13.1 Chroma 初始化

class VectorStoreService:
    def __init__(self):
        self.vector_store = Chroma(
            collection_name="enterprise_agent",   # 集合名称

            chunk_overlap=20,
            separators=["\n\n", "。", ".", "？", " ", ""],

13.2 文档加载流程（load_document）

  1. 扫描文件夹：从 data/enterprise/ 目录列出所有 .txt 和 .pdf 文件
  2. MD5 去重检查：计算文件 MD5，与已处理的 MD5 列表比对，跳过重复文件
  3. 文档加载：根据文件类型调用 PyPDFLoader 或 TextLoader
  4. 文本切分：使用递归字符分割器将长文档切为语义块
  5. 向量存储：调用 Chroma.add_documents() 将文本块向量化并持久化
  6. 记录 MD5：将已处理文件的 MD5 写入 md5_enterprise.txt

13.3 MD5 去重机制

def check_md5_hex(md5_for_check: str):
    if not os.path.exists("md5_enterprise.txt"):
        return False    # 文件不存在，说明从未处理过任何文档

    with open("md5_enterprise.txt", "r") as f:
        for line in f.readlines():
            if line.strip() == md5_for_check:
                return True  # 已处理过，跳过
        return False       # 未处理过，需要加载

这避免了重复上传同一个文件时浪费计算资源重复做 embedding。

13.4 retriever 的创建

def get_retriever(self):
    return self.vector_store.as_retriever(search_kwargs={"k": 3})

as_retriever() 返回的是一个 LangChain BaseRetriever 接口对象， 调用 retriever.invoke(query) 就会自动完成"查询向量化 → 相似度搜索 → 返回 Document 列表"的全流程。

14. 代码详解：模型工厂

model/factory.py 使用抽象工厂模式统一管理模型的创建。

14.1 抽象工厂模式

class BaseModelFactory(ABC):
    @abstractmethod
    def generator(self) -> Optional[Embeddings | BaseChatModel]:
        pass

class ChatModelFactory(BaseModelFactory):
    def generator(self):
        return ChatTongyi(model="qwen-max")

class EmbeddingsFactory(BaseModelFactory):
    def generator(self):
        return DashScopeEmbeddings(model="text-embedding-v4")
为什么用工厂模式？
虽然当前只使用了通义千问一家模型提供商，但工厂模式为未来扩展留下了清晰的结构。 如果要切换为 OpenAI 或本地模型，只需新增一个工厂子类，其他代码完全不受影响。

14.2 模块级单例

# 模块加载时立即创建，全局共享同一实例
chat_model = ChatModelFactory().generator()
embed_model = EmbeddingsFactory().generator()

Python 的模块导入机制保证了 chat_model 和 embed_model 在整个应用中只创建一次。 当 app.py 在运行时切换模型时，直接修改这些模块变量（如 model_factory.chat_model = ChatTongyi(...)）。

15.4 file_handler.py — 文件处理

utils/file_handler.py

def get_file_md5_hex(filepath: str):
    md5_obj = hashlib.md5()
    chunk_size = 4096  # 4KB 分片读取，避免大文件撑爆内存
    with open(filepath, "rb") as f:  # 必须二进制模式
        while chunk := f.read(chunk_size):
            md5_obj.update(chunk)
    return md5_obj.hexdigest()

使用了 Python 3.8+ 的海象运算符 :=，在 while 循环中同时赋值和判断。

16. 完整执行流程追踪

以用户输入 "我还有多少年假？" 为例，追踪整个系统的完整执行路径：

16.1 阶段 1：前端接收

Streamlit st.chat_input("向企业知识库提问") │ ▼ stream_agent_response("我还有多少年假？") │ ├─► sync_runtime_settings() — 同步模型/Top-K/Temperature 配置 ├─► retrieve_sources() — 预检索知识库（供 UI 展示） └─► agent.execute_stream() — ★ 进入 Agent 核心

16.2 阶段 2：Agent 推理（第一轮）

@before_model → log_before_model: "即将调用模型，带有 1 条消息" │ ▼ @dynamic_prompt → report_prompt_switch: context["report"]=False → 返回 main_prompt │ ▼ LLM 接收到完整 Prompt（系统提示词 + ReAct 格式 + 用户问题） │ ▼ LLM 输出: Thought: 用户想知道自己的年假余额，我需要先获取员工ID Action: get_employee_id Action Input: ""

16.3 阶段 3：工具执行

LangGraph 解析 LLM 输出 → 识别到 Action: get_employee_id │ ▼ @wrap_tool_call → monitor_tool: - logger.info("执行工具：get_employee_id") - handler(request) → 调用 get_employee_id() - random.choice → "E1003" - logger.info("工具 get_employee_id 调用成功") - 返回 ToolMessage(content="E1003")

16.4 阶段 4：Agent 推理（第二轮）

AgentState 现在有 3 条消息: [HumanMessage, AIMessage(Action:get_employee_id), ToolMessage("E1003")] │ ▼ LLM 接收到更新后的历史: Thought: 已获得员工ID E1003，现在查询年假余额 Action: fetch_employee_records Action Input: E1003,leave_balance

16.5 阶段 5：工具再次执行

monitor_tool: - logger.info("执行工具：fetch_employee_records") - handler → load_employee_records() → 从 CSV 读取数据 - 查找 employee_records["E1003"]["leave_balance_days"] - 返回 ToolMessage("剩余年假：12 天。") AgentState: [HumanMessage, AIMessage, ToolMessage, AIMessage, ToolMessage]

16.6 阶段 6：最终回答

LLM 接收 5 条消息的历史，输出: Thought: 我已获得年假数据，可以给出最终答案 Final Answer: 根据系统记录，您当前剩余年假为 12 天。 │ ▼ LangGraph 检测到 Final Answer → 停止循环 │ ▼ AgentExecutor.stream() yield {"output": "根据系统记录，您当前剩余年假为 12 天。"} │ ▼ Streamlit st.write_stream() → 逐字符渲染 → 用户看到打字机效果
关键技术点
整个过程 LLM 被调用了 2 次（第一轮推理 + 第二轮推理），工具被调用了 2 次（get_employee_id + fetch_employee_records）。 如果加上 RAG 相关的问题（如制度问答），内部还会多一次 rag_summarize 的 LLM 调用。

17. Prompt 工程设计

本项目共有 3 个 Prompt 文件，各司其职。

17.1 main_prompt.txt — Agent 系统提示词

定义了 Agent 的角色、目标、工具使用规则和输出规范。核心设计思路：

  • 角色锚定："你是企业内部知识库 Agent 助手"——明确身份边界
  • 场景分级：制度类→rag_summarize，个人数据→get_employee_id→fetch，周报→四步固定流程
  • 安全护栏："不编造制度条款、审批结果、报销状态"——防止幻觉
  • 兜底策略："明确说'不确定'，并建议联系对应部门"

17.2 rag_summarize.txt — RAG 总结 Prompt

这是一个填空式模板，在运行时通过 PromptTemplate.from_template() 填充变量：

用户问题：{input}
参考资料：{context}
回答要求：只使用参考资料中的信息回答。如果不足，说"不确定"。

17.3 report_prompt.txt — 周报生成 Prompt

定义了周报的四段式输出格式：本周工作概览 → 关键产出 → 问题与风险 → 下周计划。要求语言"适合直接复制到工作汇报中"。

Prompt 设计的核心原则
1. 约束优先于自由：明确告诉 LLM 不能做什么（不编造）
2. 结构化输出：提前定义输出格式，减少不确定性
3. 渐进式复杂度：简单任务直接回答，复杂任务分步执行
4. 边界声明：知识盲区时主动承认"不确定"而非胡编

18.3 关键技术决策

  • 选择 LangGraph 的 create_react_agent 而非手写 ReAct 循环：获得中间件支持、自动状态管理、流式输出
  • 选择 Chroma 作为向量数据库：轻量级、无需独立服务进程、适合演示和学习项目
  • 选择通义千问（DashScope）作为模型提供商：中文能力强、API 成本低
  • 选择 Streamlit 作为前端：纯 Python、快速开发、内置聊天组件
  • YAML 外部配置 + 模块级单例：兼顾灵活性和简单性，适合中小规模项目
  • MD5 去重机制：避免重复上传同一文件时重复 embedding，节省计算资源
